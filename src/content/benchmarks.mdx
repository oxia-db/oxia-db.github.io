# Benchmarks

---

This page is the comprehensive benchmark report for Oxia: full methodology, system
configurations, complete results across cluster sizes, value sizes, and coordination
primitives, and links to the raw data. A condensed subset of these results appears in the
Oxia paper; this page is the extended record and is kept up to date as new runs land.

All results were collected with the open-source, service-agnostic
[benchmark framework](https://github.com/oxia-db/benchmark), which drives identical
workloads against Oxia, ZooKeeper, and etcd using asynchronous clients so that the target
systems — not the load generators — are the bottleneck.

## Methodology

Two rules govern every comparison on this page:

1. **Contract-equivalent recipes.** Where a system cannot express an operation natively
   (e.g., atomic `+N` sequence assignment), we implement the closest contract-equivalent
   recipe with its native primitives (e.g., ZooKeeper sequential znodes, etcd
   transactions) and report the extra round-trips alongside the numbers, together with any
   remaining semantic difference.
2. **Documented limits are measurements.** Where a system cannot hold the state or sustain
   the load at all, we run it to its vendor-documented operational limit and report that
   boundary as a result, with the configuration disclosed.

## Throughput scalability

Four workloads, chosen to stress different paths: insert-only with ordered keys (bulk
metadata creation), 100% updates with Zipf-distributed keys (cursor/offset traffic), a
50/50 read-update mix, and 100% reads (routing lookups). Sustained throughput in
operations per second:

| Workload | ZooKeeper-3 | etcd-3 | oxia-3 | oxia-6 | oxia-12 |
|---|---|---|---|---|---|
| Insert-only (ordered keys) | 18.1k | 11.9k | 115k | 230k | 369k |
| Updates, Zipf (100% writes) | 0.5k | 8.6k | 85k | 197k | 331k |
| Mixed, Zipf (50% reads / 50% updates) | &lt;0.1k | 11.0k | 120k | 290k | 453k |
| Reads, Zipf (100% reads) | 4.4k | 14.6k | 120k | 320k | 840k |

Oxia's throughput is stable for the duration of every run. Under sustained writes, both
ZooKeeper and etcd degrade as accumulated state grows.

## Coordination primitives

These benchmarks exercise the coordination primitives directly.

**Sessions and ephemerals** — directly comparable across all three systems (ZooKeeper
sessions + ephemerals, etcd leases + lease-attached keys): session churn rate, heartbeat
overhead on foreground traffic, and ephemeral-cleanup latency from expiry to visible
deletion.

**Notifications** — end-to-end delivery latency from write acknowledgment to subscriber
delivery, across subscriber counts and write rates. etcd watches (resumable by revision)
are compared directly; ZooKeeper's single-fire watches are driven in their standard
watch–fire–re-register–re-read loop, with re-read amplification reported.

**Atomic sequences** — two experiments: a single hot sequence (one shard's assignment
ceiling, across concurrent-writer counts) and many sequences spread across shards
(horizontal scaling). Compared against contract-equivalent recipes: ZooKeeper sequential
znodes (create + set + eventual delete, `+1` only) and etcd transactional
read-modify-write on a counter key, each implementing the same job — assign a batch of N
offsets, record the index entry, notify consumers.

## Capacity: beyond-memory state

Single-RSM coordination services hold the entire dataset in memory on every node; Oxia
stores shard state in per-replica LSM trees. This experiment grows the dataset until the
baselines reach their documented operational limits — etcd's recommended storage quota and
ZooKeeper's JVM-heap-resident data tree — and reports sustained throughput as state grows,
with each system's termination point annotated.
