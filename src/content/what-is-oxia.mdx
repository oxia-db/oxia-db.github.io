# What is Oxia?

Oxia is a horizontally scalable metadata store and coordination system — the same programming model as
ZooKeeper, Etcd, or Chubby (linearizable keys, sessions, ephemerals, notifications, compare-and-swap), but
sharded across many nodes so that throughput and capacity grow with the cluster instead of being bounded by a
single leader. It is open source and a [CNCF Sandbox project](https://www.cncf.io/projects/oxia/).

## The problem

ZooKeeper, Etcd, and Consul all share one architecture: a single replicated state machine driven by Paxos or
Raft. Every node holds the full dataset, every write passes through the leader, and periodic snapshots of the
full state machine stall I/O. At scale, this shows up as:

- A hard ceiling of a few gigabytes of metadata — full-dataset snapshots become prohibitive past that.
- Write throughput capped by one leader, regardless of cluster size.
- Long tail latencies from snapshot I/O and GC pressure.

For large systems this is a real problem. Apache Pulsar, for example, was forced to keep cursor positions in
BookKeeper ledgers — separate from the rest of its metadata — because ZooKeeper could not sustain the
required write rate. That split metadata across two systems and doubled the operational surface area.

Sharded key–value stores (FoundationDB, TiKV, CockroachDB, Redis Cluster) solve the scaling problem, but they
lack the coordination primitives — sessions, ephemerals, ordered notifications, atomic sequencing — that
distributed systems need to operate. Builders end up reimplementing those primitives by hand on top of a
generic KV store, or stitching a coordination service together with a KV store and paying for both.

## Oxia's approach

Oxia partitions each [namespace](./features/namespaces.mdx) into independently replicated
[shards](./architecture/logical-architecture.mdx). Each shard has its own leader, followers, write-ahead log,
and LSM-backed state ([Pebble](https://github.com/cockroachdb/pebble)). Shards are assigned and re-elected
by a lightweight coordinator that is **off the data path**; the coordinator stores its own state in
Kubernetes (as a ConfigMap) or in a built-in Raft group when Kubernetes is not available.

Each shard runs a log-replication protocol that is deliberately **not Raft**. Raft fuses election logic and
log matching into the data-path code, which makes the hot path more complex than it needs to be. Oxia
instead separates log replication (optimized for throughput) from leader election and recovery (optimized
for correctness) — the same principle that [Apache BookKeeper](https://bookkeeper.apache.org/docs/development/protocol)
and Delos use. Safety is enforced by the coordinator: it alone advances each shard's *term*, fences stale
leaders before electing new ones, and drives recovery. Split-brain is prevented by construction, not by
vote counting.

On top of that sharded substrate, Oxia exposes coordination primitives that generic KV stores don't have:
sessions with [ephemeral records](./features/ephemerals.mdx),
[reliably ordered notifications](./features/notifications.mdx), and atomic multi-dimensional
[sequence assignment](./features/sequence-keys.mdx) fused with a `Put`. These are the building blocks of
leader election, fencing, service discovery, cache coherence, and streaming offset assignment.

## What you get

- **Per-key linearizability**, with read-after-write and durable-after-ack semantics.
- **Horizontal scale:** throughput and capacity grow by adding storage nodes. Hundreds of gigabytes of
  metadata per cluster, no full-dataset snapshots.
- **Coordination primitives:** sessions, ephemerals, compare-and-swap, range scans, and reliable
  notifications that clients can resume from an acknowledged offset.
- **Atomic sequences** with arbitrary `+N` deltas and multi-dimensional counters, fused with value storage
  — the primitive that lets [StreamNative's Ursa](https://streamnative.io/products/ursa) assign offsets
  from any availability zone without per-partition leaders.
- **Cloud-native operation:** deploy with a single [Helm chart](./deployment/kubernetes/helm.mdx) on
  Kubernetes, or run self-contained with a built-in Raft coordinator
  [on bare metal](./deployment/bare-metal.mdx). Shards split and merge automatically as load changes.
- **Observability:** OpenTelemetry metrics out of the box; Grafana dashboards in the repo.
- **Formally verified:** the single-shard protocol is modeled in [TLA+](./correctness/tla+.mdx); the
  production code is continuously validated with [Maelstrom](./correctness/maelstrom.mdx) (Jepsen-style
  linearizability checks) and Chaos Mesh failure injection.
- **Clients in [Go](./clients/go.mdx), [Java](./clients/java.mdx), and [Python](./clients/python.mdx).**

## What Oxia is *not*

- **Not globally linearizable across shards.** Each shard is linearizable on its own; cross-shard ordering
  is the application's responsibility. For coordination workloads this is almost always what you want —
  fencing, leader election, and offset assignment all operate on a single key or key range.
- **Not a transactional database.** There are no multi-key transactions that span shards.
- **Not aimed at small deployments** where a 3-node ZooKeeper or Etcd is already sufficient — Oxia's
  benefit is at scale.

## How Oxia compares

|                           | ZooKeeper / Etcd         | Sharded KV (FoundationDB, TiKV, …) | Oxia                                                   |
|---------------------------|--------------------------|------------------------------------|--------------------------------------------------------|
| Scaling model             | Single RSM               | Sharded                            | Sharded                                                |
| Write throughput ceiling  | Bounded by one leader    | Scales horizontally                | Scales horizontally                                    |
| Metadata capacity         | ~GBs (snapshot-bounded)  | Scales horizontally                | Hundreds of GBs, LSM-backed                            |
| Consensus                 | Paxos / Raft (data-path) | Multi-range Raft                   | Log replication + external coordinator                 |
| Sessions / ephemerals     | Yes                      | No                                 | Yes                                                    |
| Ordered notifications     | Watches (single-fire)    | No                                 | Yes — reliable and resumable                           |
| Atomic sequencing         | `+1` znodes only         | No                                 | Arbitrary `+N`, multi-dimensional, fused with `Put`    |
| Full-dataset snapshots    | Yes (I/O stalls)         | No                                 | No (WAL + LSM)                                         |

## Performance

Measured with Oxia's [open-source benchmark tool](https://github.com/oxia-db/benchmark) running YCSB-style
workloads on Kubernetes (AWS `i7ie.xlarge` instances, replication factor 3, 10M-key dataset, 64-byte
values). ZooKeeper and Etcd are shown with 3-node ensembles — adding replicas to a single-RSM system
improves availability but not throughput. Oxia is measured at 3, 6, and 12 storage nodes to illustrate
scaling.

| Workload                                    | ZooKeeper (3) | Etcd (3)   | Oxia (3)    | Oxia (6)    | Oxia (12)   |
|---------------------------------------------|---------------|------------|-------------|-------------|-------------|
| Inserts — 100% write, ordered keys          | ~33k ops/s    | ~25k ops/s | ~120k ops/s | ~250k ops/s | ~380k ops/s |
| YCSB X — 100% updates, Zipf skew            | ~1.7k ops/s   | ~17k ops/s | ~93k ops/s  | ~171k ops/s | ~366k ops/s |
| YCSB A — 50/50 reads/updates, Zipf          | ~33 ops/s     | ~11k ops/s | ~120k ops/s | ~296k ops/s | ~459k ops/s |
| YCSB C — 100% reads, Zipf                   | ~4.6k ops/s   | ~15k ops/s | ~120k ops/s | ~320k ops/s | ~841k ops/s |

Two things stand out:

- **Throughput scales nearly linearly** as storage nodes are added. Reads on a 12-node cluster approach
  one million operations per second.
- **Skewed (Zipf) workloads do not collapse Oxia.** Sharding places hot keys on different shard leaders,
  so a few popular records don't bottleneck the cluster the way they do in a single-RSM design.
